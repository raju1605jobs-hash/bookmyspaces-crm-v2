# PRODUCTION_SCORECARD.md

Scores are 0 (not production-ready at all) to 10 (fully production-ready, no known gaps). Every score is explained with direct references to the audit files. These scores reflect the state of the codebase AS AUDITED — before any of the roadmap's fixes are applied.

| Area | Score | Basis |
|---|---|---|
| **Authentication** | **1/10** | `AUTH_AUDIT.md`: the active middleware (`src/middleware.ts`) unconditionally allows every request with a comment stating "TEMP: allow everything while stabilizing app"; no `layout.tsx` anywhere performs an auth check; only 4 of 31 API routes check a session at all. Real Supabase Auth is correctly wired for login/logout/OAuth (the one bright spot), but nothing enforces that anyone actually be logged in to use the app. |
| **Authorization** | **1/10** | `AUTH_AUDIT.md`: exactly one route (`admin/users`) attempts a role check, and that check compares the wrong field (`user.role`, the GoTrue auth role) against values (`'admin'`, `'manager'`) that only exist in the separate, never-joined `user_profiles.role` column. No other route has any permission model at all. |
| **Database** | **4/10** | `DATABASE_AUDIT.md`: the schema that DOES exist is well-structured — proper foreign keys, check constraints, triggers, indexes (including HNSW vector indexes), and idempotent migration patterns. But 5 tables/views (`active_users_view`, `invoices`, `lead_imports`, `payments`, `user_profiles`) and 1 column (`leads.lead_stage`) used by live routes don't exist in any of the 9 migrations — a database built strictly from this repo cannot run large parts of the application. One RLS gap (`notification_settings`) and one unscoped anon-write policy (`conversations`) further reduce the score. |
| **Security (overall)** | **2/10** | Combination of Authentication (1/10) and Authorization (1/10) above, plus: `DEPLOYMENT_AUDIT.md` — no security headers configured, Server Actions CSRF origin check explicitly disabled for all origins; `ROUTES.md`/`AUTH_AUDIT.md` — WhatsApp webhook accepts unsigned payloads; `ROUTES.md` — zero input validation on any of 31 routes. |
| **API** | **4/10** | `ROUTES.md`: consistent try/catch error-handling pattern across all 31 routes is a genuine positive, and the route-to-table mapping is sensible and traceable. But zero routes validate input (zod is declared, unused), 26 of 31 have no auth check, and 2 additional route files are misnamed and silently dead (never registered by Next.js). |
| **Frontend** | **6/10** | `PAGE_INVENTORY.md`: all 12 in-scope pages are genuinely implemented, no stubs or "coming soon" placeholders found — a real positive for a project this size. `COMPONENT_INVENTORY.md`: offset by one 932-line God Component (`HotLeadDashboard.tsx`) with 16 inline sub-components and no formal prop typing; `DEPLOYMENT_AUDIT.md`: two divergent Tailwind configs create uncertainty about what's actually rendering. |
| **Backend** | **5/10** | `LIBRARY_INVENTORY.md`/`MODULE_INVENTORY.md`: the business logic that IS wired in (lead scoring, proposal intelligence, AI chat/RAG, Google Sheets sync) is clean, typed, and TODO-free. But three significant subsystems are dead code with zero importers — the entire alternate WhatsApp conversation state machine (~1020 LOC), the escalation engine, and the follow-up cadence engine — meaning a meaningful fraction of the backend's apparent capability does not actually run. |
| **Performance** | **5/10** | `ROUTES.md`/`DEPLOYMENT_AUDIT.md`: no rate limiting at the route level (exists only in an unused-by-most-routes library helper), a blanket `no-store` cache header on every API response forecloses any edge caching, and proposal PDF generation depends on a live Google Fonts network fetch at render time (`LIBRARY_INVENTORY.md`). No confirmed N+1 query patterns or other acute performance defects were found in this pass, but performance was not load-tested. |
| **Scalability** | **5/10** | Architectural observation from `ROUTES.md`: 28 of 31 routes share a single service-role database trust tier with no RLS-based isolation, which works today but will need deliberate redesign (ISS-006 in the issue register) if the product ever needs per-tenant or per-role data isolation beyond what a single service-role key can express. |
| **Maintainability** | **4/10** | `CODE_METRICS.md`: 77 explicit `any`-type escapes against a `strict: true` tsconfig, 89 `console.*` calls repo-wide, no centralized environment configuration (18 vars read directly across 21 files). `LIBRARY_INVENTORY.md`: business contact info and pricing hardcoded across 6+ files instead of one config module. |
| **Code Quality** | **5/10** | `DEPLOYMENT_AUDIT.md`: `strict: true` is correctly set in `tsconfig.json`, and the consistent error-handling pattern across all routes is real evidence of care. Offset by the `any`-erosion above, zero ESLint rules beyond `next/core-web-vitals` defaults (no `no-explicit-any`, no `no-console`), and duplicate code (two Meta API implementations, two `CookieItem` interfaces, two OAuth callback implementations). |
| **Documentation** | **3/10** | `CODE_METRICS.md`/`env_usage.csv`: `.env.example` actively mismatches what the code reads in multiple places (`NEXT_PUBLIC_SITE_URL` vs `NEXT_PUBLIC_APP_URL`, `WATI_VERIFY_TOKEN` vs `WHATSAPP_WEBHOOK_VERIFY_TOKEN`), and two required variables (`WHATSAPP_ACCESS_TOKEN`, `WHATSAPP_PHONE_NUMBER_ID`, `CRON_SECRET`) aren't documented there at all — a new deployer following the example file will not get a working WhatsApp integration or cron job. |
| **Deployment** | **4/10** | `DEPLOYMENT_AUDIT.md`: `vercel.json` is genuinely well-configured (correct region, per-route function timeouts, a working cron schedule). But there is no startup environment validation, no setup wizard, and no fail-fast behavior — combined with the documentation gaps above, this directly blocks the "install and immediately start using it" goal from the original project brief. |
| **Testing** | **0/10** | `DEPLOYMENT_AUDIT.md`: zero test files, zero test framework in dependencies, no `test` script in `package.json`. This is the most unambiguous score on this scorecard — there is categorically nothing to partially credit. |
| **Business Logic** | **5/10** | The core CRM loop that IS reachable (lead capture → AI scoring → kanban → proposal → PDF → payment → invoice/receipt) is genuinely implemented per `PAGE_INVENTORY.md`/`ROUTES.md`. But `MODULE_INVENTORY.md`/`PROJECT_DISCOVERY.md` show the automation layer the CRM's brief promises — automated escalation, automated follow-up cadence, automated daily AI summary, automated proposal-email delivery — is either disconnected dead code or (for email) not actually implemented server-side at all. |
| **OVERALL PRODUCTION READINESS** | **3/10** | Weighted toward Authentication, Authorization, Database, and Testing because a CRM handling customer PII and payment data cannot be called production-ready with zero access control and zero automated verification, regardless of how complete the feature surface looks. The underlying feature set and code craftsmanship (where actually wired in) are genuinely above-average for a project this size — this is not a "start over" situation, it's a "close the gaps before onboarding paying customers" situation, consistent with the roadmap in `IMPLEMENTATION_PHASES.md`. |

## Why 3/10 and not lower

The audit found zero stub pages, zero fabricated/placeholder features, clean and typed business logic wherever it's actually wired in, a well-evolved (if incomplete) database schema with proper constraints and indexes, and a consistent architectural pattern across all 31 API routes. This is meaningfully better than a typical early-stage codebase with this security posture — the gaps are concentrated and fixable (see `IMPLEMENTATION_PHASES.md`), not systemic across every dimension.

## Why 3/10 and not higher

Authentication, Authorization, and Testing scoring 0-1/10 each are not "areas for improvement" — they are the specific conditions under which real customer data (leads, proposals, payment records) is currently exposed to any unauthenticated visitor who finds the app's URL, with no automated way to verify a fix doesn't silently break something else. No amount of feature completeness offsets that combination for a "production ready" verdict.
